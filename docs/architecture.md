# App-Core 应用层架构说明

## 1. 目标与边界

App-Core 的目标是为多人协作的 Linux C 应用提供一套可执行的架构约束：模块能被发现、依赖能被验证、
启动和停止可回滚、通信契约可版本化、并发资源可回收、故障可观测。它不是 Linux 的替代品，也不尝试
实现调度器、网络栈、块层、VFS、驱动模型或安全隔离边界。

适合承载工业网关、边缘设备、媒体服务、协议转换、采集存储、设备管理、后台 daemon 和大型单进程
应用。硬实时控制、安全认证系统、强进程隔离平台和天然分布式系统仍需要专用架构补充。

## 2. 分层

```text
业务模块（网络 / 存储 / 媒体 / 设备 / 算法）
        │  类型化服务 / 事件 / 数据通道
应用治理层（组件生命周期、运行时、日志、状态与错误）
        │  pthread / fd / poll / signalfd / eventfd
Linux 用户态 API 与成熟协议、数据库、媒体和安全库
        │
Linux 内核（调度、网络、存储、内存、驱动、安全）
```

`core/` 只能包含与业务无关的机制；业务事件 ID、协议结构、服务操作表和数据格式必须由所属模块定义。
依赖方向只能从上层指向稳定的下层接口，不允许通过全局变量或私有头跨层访问实现。

## 3. 借鉴 Linux 的部分

- 链接段静态发现组件，类似内核 initcall/section 思路。
- 显式阶段、依赖和确定性排序，避免依靠源文件链接顺序。
- 引用计数控制服务、通道和 buffer 生命周期。
- 有界队列和明确背压，不以无限内存掩盖生产消费失衡。
- 文件描述符化的信号和退出通知，统一进入 `poll` 控制流。
- 机制与策略分离：core 提供 FAIL/DROP_OLDEST，业务决定何时使用。

与 Linux 内核的关键差异是：App-Core 不提供权限边界、进程隔离、抢占调度、设备模型、sysfs/procfs、
RCU、内核内存分配器或完整驱动基础设施。模块位于同一地址空间，一个越界写仍可破坏整个进程。

## 4. 生命周期状态机

组件状态从 DISCOVERED 进入 INITIALIZING、INITIALIZED、STARTING、RUNNING；关闭时进入 STOPPING 和
STOPPED。回调失败进入 FAILED，依赖不可用进入 SKIPPED。框架只启动依赖已经 RUNNING 的组件，并按实际
成功顺序逆序停止和释放。

required 表示该能力是应用成立的前提；optional 表示失败可降级。optional 失败不是被吞掉，而是保留
错误码和状态，由 `sys_core_get_status()` 暴露给健康检查或管理面。

## 5. 三种通信平面

`sys_service` 是低频控制面，适合查询、配置和同步命令。调用方持有临时引用，注销方先停止接收新引用，
再等待在途调用归零。

`sys_event` 是消息面。同步发布用于需要错误反馈和停机握手的通知；异步发布会深拷贝小 payload 并进入
固定容量队列。事件总线停机时会等待同步与异步在途回调结束，避免释放订阅对象后仍被访问。

`sys_channel` 是大流量数据面。队列只传递 buffer 指针，payload 在预分配池中复用。每个 buffer 携带
type、version、generation、sequence 和 timestamp，消费者可拒绝过期流或不兼容格式。

## 6. 性能模型

- 服务调用增加一次注册表锁和引用计数，适合控制路径，不应放在逐字节或极高频内循环。
- 同步事件成本与订阅者数量线性相关；异步事件会分配并复制 payload，只适合小消息。
- 数据通道发送/接收各使用一次互斥锁和条件变量信号，但不复制 payload；对视频帧、文件块等大对象，
  框架开销通常远小于 I/O、编解码和协议处理成本。
- 日志格式化和默认 stderr 输出可能成为热路径瓶颈，生产环境应设置级别或接入异步 sink。
- 当前通道是通用 MPMC 有界队列，不承诺 lock-free。只有基准证明锁竞争成为瓶颈时，才应为特定场景
  增加 SPSC/批量 API，而不替换默认的易读实现。

## 7. 尚需由具体项目补充的能力

- 配置热更新、持久化和密钥管理。
- watchdog、自动重启、崩溃转储和 systemd 集成。
- 指标导出（Prometheus、OpenTelemetry 等）与生产日志后端。
- 权限下降、seccomp、namespace、容器或多进程故障隔离。
- 网络连接池、协议编解码、数据库迁移、存储一致性等领域组件。
- 长周期兼容性测试、性能基线和面向目标硬件的容量规划。

这些能力应作为独立模块接入，而不是继续扩大 core。core 的稳定性来自接口少、职责窄和行为可测试。
